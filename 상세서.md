# 의료 AI 서비스 MVP UI/UX 상세서 (Web · Mobile Responsive)

- 문서 버전: v0.1 (MVP)
- 작성일: 2026-01-16
- 대상: 기획/디자인/프론트엔드/백엔드 공통 참고

---

## 0. 전제/가정(변경 가능)

- **서비스 형태**: 의료 데이터(문진/검사결과 등)를 입력하면 **상용 AI 모듈/API**가 분석 결과를 제공하는 B2B/B2B2C형 MVP(웹 기반, 모바일 대응).
- **사용자 역할**: `의료진(또는 상담사)` / `관리자(Admin)` 중심. (환자 직접 사용은 2차 범위로 두되, 화면 구조는 확장 가능하게 설계)
- **핵심 기능(최소)**: 케이스 생성 → 데이터 입력/업로드 → AI 분석 요청/진행상태 → 결과 조회/공유(PDF) → 케이스 관리.
- **금지/주의**: UI 문구는 “진단/처방”이 아니라 **“참고/보조”** 표현을 기본으로 하며, 결과 화면에 **의료 책임 고지**를 고정 노출.

---

## 1. UX 목표(원칙)

1. **신뢰**: 의료 서비스 톤(차분함/정확함), 근거/버전/시간을 명확히 노출
2. **안전**: 위험도(High/Medium/Low) 강조 + “응급” 시 행동 유도(연락/내원) UI 제공
3. **최소 입력**: MVP는 “필수 최소 항목”만 요구, 나머지는 선택 입력으로 단계화
4. **투명성**: AI 생성임을 명확히 표시하고, “확신도/한계/주의사항” 섹션을 표준화
5. **협업**: 케이스 단위로 메모/상태/담당자/공유 내역이 남는 구조

## 2. 정보 구조(IA) & 내비게이션

### 2.1 메뉴 구조(권장)

- **대시보드**
  - 케이스 목록(검색/필터/정렬)
  - 내 작업(담당 케이스)
- **새 케이스**
  - 문진/자료 업로드
  - AI 분석 요청
- **리포트(옵션)**
  - 사용량/성능 지표(관리자)
- **설정(관리자)**
  - 사용자/권한
  - AI 연동 설정(API Key, 엔드포인트, 모델 버전)
  - 감사로그/접근기록

### 2.2 내비게이션 패턴

- **Desktop(≥1024px)**: 좌측 사이드바(고정) + 상단 헤더(고정)
- **Tablet(768~1023px)**: 사이드바 축소(아이콘) 또는 햄버거 드로어
- **Mobile(<768px)**: 상단 헤더 + 하단 탭바(최대 4개) / 설정은 더보기 또는 드로어

### 4.2 공통 레이아웃 규격

  - max-width: **1200px**

### 4.3 페이지 구성 규칙(템플릿)

## 5. 컴포넌트 규격(디자인/개발 공통)

> 모든 컴포넌트는 “사이즈(Height/Pad) + 상태 + 접근성”을 포함해 구현합니다.

### 5.1 버튼(Button)

#### 사이즈

| 타입 | 높이 | 패딩(좌우) | 폰트 |
|---|---:|---:|---|
| `sm` | 32px | 12px | 14px/500 |
| `md`(기본) | 40px | 16px | 14px/600 |
| `lg` | 48px | 20px | 14px/600 |

#### 변형(Variant)

- `primary`: bg `--color-primary`, text `--color-on-primary`
- `secondary`: bg `--color-secondary`, text white
- `outline`: bg transparent, border 1px `--color-border-strong`, text `--color-text`
- `ghost`: bg transparent, text `--color-text` (툴바/아이콘 근처)
- `danger`: bg `--color-danger`, text white(삭제/폐기)

#### 상태

- Hover: 배경/보더 명도 -1 단계(토큰 hover 사용)
- Disabled: bg `--color-border`, text `--color-text-subtle`, cursor not-allowed
- Loading: 버튼 높이 유지 + 스피너(16px) + 텍스트 유지 또는 “처리 중…”

### 5.2 아이콘 버튼(Icon Button)

- 크기: 40×40px(기본), 아이콘 20px
- 배경: 기본 transparent, hover 시 `--color-surface-2`
- 툴팁(Desktop): 600ms hover 시 표시

### 5.3 입력창(Input)

#### 기본 규격

- 높이: 40px(기본), 큰 입력 48px(온보딩/모바일 권장)
- 패딩: 좌우 12px, 텍스트 상하 중앙 정렬
- 보더: 1px `--color-border`
- 라운드: 10px
- 플레이스홀더: `--color-text-subtle`

#### 상태/검증

- Focus: 보더 `--color-primary` + 외곽 2px `--color-focus`
- Error: 보더 `--color-danger`, 하단 helper 텍스트(12px, `--color-danger`)
- Success(선택): 보더 `--color-success`(주로 폼 제출 후)

#### 라벨/도움말 규칙

- 라벨: 입력 위, `Caption`(12px) / 필수는 `*` 표기
- 도움말: 라벨 아래 또는 입력 아래(일관성 유지), 최대 2줄
- 오류 문구는 “무엇이/왜/어떻게”를 포함: 예) “생년월일 형식이 올바르지 않습니다(YYYY-MM-DD).”

### 5.4 텍스트 영역(Textarea)

- 최소 높이 96px, resize는 기본 off(레이아웃 안정), 필요 시 “확장” 버튼 제공
- 글자 수 카운터(옵션): 우하단 12px, 0/500 형태

### 5.5 셀렉트/드롭다운(Select/Dropdown)

- 트리거: Input과 동일 규격(높이/보더/라운드)
- 드롭다운: 최대 높이 280px, 스크롤, 항목 높이 40px
- 키보드: ↑↓ 이동, Enter 선택, Esc 닫기

### 5.6 카드(Card)

- 라운드 12px, 배경 `--color-surface`, 보더 1px `--color-border`, 그림자 `--shadow-sm`
- 패딩: 16px(기본), 내부 요소 간격 12px
- 카드 헤더(선택): 제목(H2/H3) + 우측 액션(아이콘 버튼)
- 클릭형 카드(리스트): hover 시 `--shadow-md` + 보더 `--color-border-strong`

### 5.7 모달(Modal)

- 오버레이: `--color-overlay`
- 폭:
  - `sm` 360px(간단 확인)
  - `md` 560px(폼/설정)
  - `lg` 720px(미리보기/결과 공유)
- 최대 높이: 80vh, body 스크롤
- 헤더: 타이틀(H2) + 닫기(X)
- 푸터: 우측 정렬 버튼 2개(취소 Outline, 확인 Primary)
- 닫기 규칙: `ESC` 가능, 오버레이 클릭 닫기 여부는 케이스별(데이터 손실 위험 시 비활성)

### 5.8 토스트/배너(Toast/Banner)

- Toast: 우하단(Desktop) / 상단(모바일) 스택, 5초 자동 종료(중요 오류 제외)
- Banner(페이지 상단 고정형): 네트워크 오류/권한/연동 실패 등 “상태 유지형” 메시지

### 5.9 테이블/리스트(Table/List)

- 헤더 높이 44px, 바디 행 높이 48px
- 첫 컬럼은 “식별 정보(케이스ID/환자 요약)” 고정 권장
- 모바일은 테이블을 카드 리스트로 변환(행당 2~3줄 요약)
- 정렬: 컬럼 헤더 클릭(가능 시), 정렬 아이콘 표시

### 5.10 배지/태그(Badge/Tag)

- 높이 24px, 패딩 8px, 폰트 12px/500
- 상태 배지 예시:
  - `대기`(회색), `분석중`(Info), `완료`(Success), `실패`(Danger)
  - 위험도 배지: Low/Medium/High (3.1 표준 색상 사용)

### 5.11 스텝퍼(Stepper) — 케이스 생성 플로우

- 단계: 4단계 권장(기본정보 → 문진 → 자료첨부 → 확인/요청)
- 현재 단계 강조(Primary), 완료 단계 체크 아이콘
- 모바일은 상단 진행바(%) + 단계명 1줄로 축약

### 5.12 파일 업로드(File Upload)

- 드래그&드롭 영역: 100% 폭, 높이 120px, 점선 보더 `--color-border-strong`
- 허용 파일: 이미지(PNG/JPG) + PDF(권장) / (의료 맥락상 DICOM은 MVP 범위 밖으로 표기)
- 파일 카드: 파일명(한 줄), 크기, 업로드 상태(진행바), 삭제 아이콘
- 오류: 용량 초과/형식 불가/업로드 실패를 파일 단위로 표시

### 5.13 AI 진행상태(Processing)

- 상태 단계 표준:
  1) 요청 접수
  2) 데이터 전처리
  3) AI 분석
  4) 결과 생성
  5) 완료
- 각 단계는 아이콘 + 텍스트로 노출, 진행 중 단계는 스피너(16px)
- “재시도”는 실패 시에만 노출, “취소”는 분석 초기(1~2단계)에서만 노출(백엔드 정책과 동기화)

### 5.14 위험도/점수(Risk Indicator)

- 표기: `High/Medium/Low` + 점수(0–100) 옵션
- 권장 UI:
  - 배지(필수) + 작은 게이지 바(선택)
  - High는 상단 요약영역에 “행동 유도 CTA”(예: “응급의심 시 즉시 의료기관 안내”)를 함께 표시

## 6. 표준 상태 UI(로딩/빈 화면/오류)

### 6.1 로딩(Loading)

- 전체 페이지 로딩: 스켈레톤(카드/테이블 형태 유지)
- 버튼 로딩: 버튼 내부 스피너 + 텍스트 유지(레이아웃 흔들림 방지)
- AI 분석 로딩: 5.13 진행상태 컴포넌트 사용(단순 스피너 금지)

### 6.2 빈 화면(Empty State)

빈 화면은 아래 3요소를 고정 구성:
1) 아이콘(48px)  
2) 제목(H2) + 설명(Body)  
3) Primary CTA 1개(예: “새 케이스 만들기”)

### 6.3 오류(Error)

- 입력 오류: 필드 하단 inline error
- API/연동 오류: 페이지 상단 Banner + “재시도/문의” 액션
- 치명적 오류(권한/세션 만료): 모달로 안내 후 로그인 이동

## 7. 핵심 화면 명세(MVP)

> 각 화면은 “목적 → 레이아웃 → 구성 요소 → 상호작용/검증 → 상태” 순으로 정의합니다.

### 7.1 로그인(Login)

**목적**: 사용자 인증 및 역할 기반 접근.

**레이아웃**
- 중앙 카드(폭 360px), 배경 `--color-bg`

**구성 요소**
- 로고/서비스명
- 입력: 이메일(또는 ID), 비밀번호
- 버튼: `로그인`(Primary)
- 보조: “비밀번호 재설정”(링크, 옵션)

**검증/상태**
- 실패: “아이디 또는 비밀번호가 올바르지 않습니다.”(Banner 또는 inline)
- 5회 실패 시: Captcha/잠금(옵션, MVP는 미포함 가능)

### 7.2 동의/주의 고지(Consent)

**목적**: 의료 AI 고지/데이터 처리 동의(감사 대응).

**구성 요소**
- 체크박스 2개(필수):
  - “본 서비스는 의료진 판단을 보조하는 참고정보입니다.”
  - “데이터 처리 및 보관 정책에 동의합니다.”
- 링크: 개인정보처리방침/이용약관(모달 또는 새 탭)
- 버튼: `동의하고 계속`(Primary)

**규칙**
- 동의 완료 전에는 케이스 생성/분석 요청 CTA 비활성
- 동의 기록: 동의일시/버전 표시(설정/프로필에서 재확인 가능)

### 7.3 대시보드/케이스 목록(Case List)

**목적**: 케이스 탐색/우선순위 처리.

**레이아웃**
- 상단 Page Header: “케이스”
- Filter Bar: 검색 + 필터 + 정렬
- 본문: Table(Desktop) / Card List(Mobile)

**테이블 컬럼(권장)**
- 케이스ID
- 환자 요약(성별/연령대/주호소 1줄)
- 상태(대기/분석중/완료/실패)
- 위험도 배지
- 생성일시 / 업데이트 일시
- 담당자(옵션)
- 액션(보기)

**필터(권장)**
- 상태, 위험도, 기간(최근 7일/30일/직접), 담당자(옵션)

**상호작용**
- 행 클릭 → 케이스 상세
- 우측 상단 CTA: `새 케이스`(Primary)

### 7.4 새 케이스 생성(4-Step)

#### Step 1) 기본정보

**필수 입력(최소)**
- 성별(Select): 남/여/기타/미상
- 연령대(Select): 0–9, 10–19 … 80+
- 주호소(Short text): 1줄(예: “흉통”, “두통”)

**선택 입력(확장)**
- 내원경로/증상 시작일, 과거력, 복용약, 알레르기(태그 입력)

#### Step 2) 문진(Questionnaire)

- 질문은 카드 1개=질문 1개(모바일 가독성)
- 질문 타입: 단일선택/다중선택/척도(0–10)/자유서술
- “모름/해당없음” 선택 가능(강제 응답 최소화)

#### Step 3) 자료첨부(Upload)

- 파일 업로드(5.12)
- 입력 가이드: “개인식별정보(주민번호 등)는 업로드하지 마세요.”(Info Banner)

#### Step 4) 확인/AI 요청(Review)

- 입력 요약 카드(수정 링크 포함)
- 버튼:
  - `AI 분석 요청`(Primary)
  - `임시저장`(Outline, 옵션)

**검증 규칙**
- 필수 3항목(성별/연령대/주호소) 누락 시 Step 이동 불가
- 업로드 파일은 선택이지만, 업로드 실패 파일이 있으면 요청 버튼 비활성 + 오류 안내

### 7.5 AI 분석 진행(Processing Screen)

**목적**: 비동기 분석 과정의 불확실성 최소화.

**구성 요소**
- 진행상태 컴포넌트(5.13)
- 예상 소요시간(가능하면): “약 20–60초”
- 버튼:
  - `백그라운드로 보내기`(케이스 목록으로 이동, 상태는 유지)
  - 실패 시 `재시도`

### 7.6 결과 화면(Analysis Result)

**레이아웃(상단 요약 + 하단 상세)**
1) 상단 Summary Card(고정 구성)
   - 위험도 배지 + 점수(옵션)
   - 한 줄 결론(예: “응급 가능성 있음(참고)”)
   - 신뢰 정보: 분석 시각, AI 버전, 입력 데이터 요약(토글)
   - 주요 CTA: `PDF 내보내기` 또는 `공유 링크 생성`(Primary 1개만)
2) 상세 영역(탭 또는 아코디언)
   - 요약(Summary)
   - 근거/관찰 포인트(Key Factors)
   - 권고사항(Next Steps)
   - 한계/주의사항(Disclosure) — **항상 노출**

**상호작용**
- 섹션별 “복사” 버튼(클립보드)
- “추가 질문하기”(옵션): 동일 케이스에 추가 입력 → 재분석(버전 증가)

**고지(필수)**
- 결과 하단 고정 문구:
  - “본 결과는 참고용이며 최종 판단은 의료 전문가의 진료를 통해 이루어져야 합니다.”

### 7.7 케이스 상세(Case Detail)

**목적**: 케이스 단위의 히스토리/협업.

**구성 요소**
- 좌(또는 상단): 케이스 요약 카드(필수 정보 + 상태/위험도)
- 우(또는 하단): 타임라인
  - 입력 이벤트(문진/업로드)
  - 분석 요청/완료/실패 로그
  - 결과 스냅샷(버전별)
  - 사용자 메모(의료진/상담사)
- 액션:
  - 상태 변경(대기→완료)
  - 담당자 지정(옵션)

### 7.8 설정(관리자) — AI 연동/사용자/로그

#### AI 연동 설정
- 필드:
  - API Base URL
  - API Key(마스킹 기본, “보기”는 재인증 필요 권장)
  - 모델/버전 선택(옵션)
  - Timeout(초)
- 버튼:
  - `연결 테스트`(Outline) → 성공/실패 토스트
  - `저장`(Primary)

#### 사용자/권한
- 역할: Admin / Clinician(의료진) / Staff(상담사) / Viewer(읽기 전용)
- 권한 예:
  - 케이스 생성/조회/다운로드
  - 설정 접근
  - 민감정보 보기(마스킹 해제)

#### 감사 로그(접근 기록)
- 필수 컬럼: 사용자, 작업(조회/다운로드/공유), 케이스ID, 일시, IP(옵션)
- 필터: 사용자/기간/작업 유형

## 8. 문구/표현 규칙(의료 AI 톤)

- 금지 표현: “확정 진단”, “처방”, “반드시”, “100%”
- 권장 표현: “가능성”, “의심”, “참고”, “추가 확인 권장”
- 사용자 불안 유발 최소화:
  - High 위험도에서는 “행동 지침”을 명확히(예: “증상이 심하면 즉시 응급실/119”)
  - Medium/Low에서는 “관찰 포인트/추가 질문” 중심

## 9. 접근성(A11y) & 품질 기준

- 대비(Contrast): 텍스트 대비 **WCAG AA** 수준(일반 텍스트 4.5:1 권장)
- 키보드: Tab 이동 순서 논리적, 모든 버튼/입력 접근 가능
- 스크린리더: 폼 라벨 연결, 오류는 aria-live(중요 오류)
- 색상만으로 의미 전달 금지: 위험도는 **텍스트(Low/Medium/High) 동반**
- 날짜/시간 표기: `YYYY-MM-DD HH:mm`(24h) 기본(로그/의료 기록에 유리)

## 10. 개발 구현 가이드(프론트/백엔드 협업 포인트)

### 10.1 화면 구현 우선순위(MVP)

1) 로그인/권한
2) 케이스 목록 + 상세
3) 새 케이스(4-Step) + 업로드
4) AI 진행상태 + 결과 화면
5) 설정(최소: AI 연동 테스트/저장)

### 10.2 API 응답에 포함되면 UX가 좋아지는 필드(권장)

- `case`
  - `id`, `status`, `risk_level`, `risk_score?`, `created_at`, `updated_at`
  - `patient_summary`(성별/연령대/주호소 1줄)
  - `assigned_to?`
- `analysis`
  - `analysis_id`, `model_name`, `model_version`, `started_at`, `finished_at`
  - `confidence?`(0–1 또는 low/med/high)
  - `summary`, `key_factors[]`, `recommendations[]`, `disclaimer`
  - `errors[]`(실패 시 사용자 노출용 메시지 포함)

### 10.3 비동기 UX를 위한 상태 코드(권장)

- `QUEUED` → `PREPROCESSING` → `ANALYZING` → `GENERATING` → `COMPLETED`
- 실패: `FAILED`(사유 코드 포함: TIMEOUT/INVALID_INPUT/UPSTREAM_ERROR)
